Google Workspace Vendor Onboarding into Apptium / ION

This document captures the full context, engineering workstreams, technical logic, and open decisions involved in stabilizing GWS as a vendor in TD SYNNEX's Apptium/ION billing and marketplace platform.

Billing Price Book Entitlement API 7 Workstreams 14 Open Questions

① Core Purpose

🎯
The Single Goal
Every change to a GWS subscription (upgrade, downgrade, plan change, cancellation, seat shift, renewal) must map to a correctly priced line item using the right entitlement context — while resellers see only their own data, and customer alternate email is captured end-to-end.
⚠️
The Central Problem
The existing Price Book architecture was built for flat product catalogs. GWS requires entitlement-level, customer-specific, time-sensitive pricing — a fundamentally different model the current platform doesn't support.

② Main Ideas, Simply Explained

📚
What is a Price Book?
The catalog that tells the system what to charge

Think of a Price Book like a restaurant menu. Today, ION has one shared menu for everyone. GWS requires each table (reseller) to have their own secret menu — with their own customer-specific prices, discounts, and deal terms. The problem: the kitchen (ION) doesn't know how to run that kind of operation yet.

The Master Price Book is what TD SYNNEX/administrators see (full pricing data). The Reseller Price Book is a filtered view — resellers only see their own customers' data, never discount terms or other resellers' info.

🔑
What is an Entitlement?
Google's record that a customer has access to a product

When a customer purchases Google Workspace, Google creates an entitlement — a unique record that says "Customer X has Y seats of Product Z under offer W, at price phase P." This is Google's source of truth for what was sold and when.

ION must read and track these entitlement records to know: what price applies today, what changed over time (upgrades, renewals, seat changes), and what credit is owed on cancellation.

🔢
What is the "Unique ID"?
A new identifier that links pricing to a specific customer's deal

Today ION prices GWS from a generic product catalog entry. The "Unique ID" is a proposed new concept — a composite key that ties a specific entitlement (customer + offer + price phase) to its exact discount percentage. This allows ION to price accurately at the day level across the full life of a subscription.

Think of it as a serial number for a customer's specific pricing deal, not just the product type.

📧
Why does Alternate Email matter?
A lost field that breaks Google customer creation

Customer.alternateEmail is a required field for creating GWS, Cloud Identity, and Chrome Enterprise customers in Google's system. It was removed when Feature 755664 (Buy Flow Redesign) shipped on 23 Jan 2026. Without it, Welcome Emails go to the end customer instead of the MSP/reseller, and the unique email constraint blocks MSPs from reusing a reseller email across multiple customers.

This is classified as a foundational defect — no new GWS customer can be properly onboarded until this field is restored.

Price Phases Discount Logic Entitlement API Unique ID

Key Concepts Explained

A pricePhase is a time-bounded segment of a subscription's pricing. Imagine a 3-year GWS deal: Year 1 might be at 30% discount, Year 2 at 20%, Year 3 at 10%. Each of those is a separate price phase. ION must know which phase is active on any given calendar day to generate correct billing. It does this by taking the subscription's createTime as the start, then counting forward using the firstPeriod value (in days) of each phase.

💡
Mental Model
Think of it like a tiered lease. If you signed a 3-year lease with decreasing discounts, the billing system must know which year you're in — day 364 vs. day 366 can mean entirely different rates.

discountPercentage is the numeric rate (e.g., .15 = 15%). discountType is a category label (e.g., RESELLER_MARGIN, SALES_DISCOUNT). Srilatha's position (Apr 26): Grouping by discountType is the better architectural fit. Any further sub-selection should be done at the row level. Brian's reply (Apr 28): Users must know the exact discount percent to apply the correct PB Price Adjust. Concrete use case: user selects all entries with .15 discountPercentage and bulk-adjusts them. Any new Unique IDs added later with that same .15 discountPercentage must automatically inherit the existing price adjustment.

⚠️
Still Unresolved — Decision Required
The two positions are not necessarily mutually exclusive: a hybrid model (group by discountType architecturally, but surface exact discountPercentage for user selection) may satisfy both sides. No formal decision has been documented yet.

The Unique ID is a proposed new concept in ION. It acts as a composite reference that links one specific customer's entitlement → offer → price-phase context → discount percentage into a single identifiable object.

  • Must resolve to an exact discountPercentage at any point in time
  • Stays stable across seat changes
  • Updates when a SKU_CHANGED or COMMITMENT_CHANGED event fires
  • New IDs with the same discount % must auto-inherit existing bulk price adjustments

Open question: Is this a database-generated surrogate key, or is it derived from Google fields (entitlementID + offerID + startDate)? This isn't decided yet.

When a customer cancels an annual GWS commitment plan, Google generates a COMMITMENT_CANCEL_ITEM charge type in the billing data. ION must issue a credit to the reseller/end-customer for the unused portion. The current bug: ION is using BigQuery's end-date value instead of calculating it independently. BigQuery's value is incorrect, causing wrong credit amounts.

Correct FormulaBilling End Date = createTime (subscription start) + 1 year Credit Calculation: Discount Factor = 1 − Price Adjust Value SaaS Customer Cost = Seller Cost ÷ Discount Factor Example: Seller Cost = −493.4 Price Adjust Value = −29.167% Discount Factor = 1 − (−0.29167) = 1.29167 Customer Cost = −493.4 ÷ 1.29167 = −697.78

This Google API returns a chronological log of every change to a customer's entitlement. ION needs this history to answer: "What was the pricing state on day X of this subscription?"

  • PAID_SUBSCRIPTION_STARTED — the original subscription start
  • SKU_CHANGED — customer switched products
  • COMMITMENT_CHANGED — term length changed
  • RENEWED — annual renewal occurred
  • RENEWAL_SETTING_CHANGED — auto-renew setting changed
⚠️
Critical Open Question
Which createTime is the authoritative start date for billing end-date calculations: the original PAID_SUBSCRIPTION_STARTED, or the most recent RENEWED or SKU_CHANGED? This must be resolved before cancel billing can be correctly implemented.

Currently, a single Price Book in ION can be shared across multiple reseller partners. This means one reseller could potentially see another reseller's discount percentages, customer domain names, and deal terms. The architectural problem: ION currently treats Price Books as completely independent of Partner-Customer associations. Implementing per-partner filtering would require ION to process all Partner-associated customer domains and their entitlements on every Price Book load — a potentially expensive operation.

Brian's Apr 28 clarification: The proposal is to filter/hide offer + entitlement entries belonging to other partners on load. The SKU must remain easily identifiable alongside the Unique ID at all times.

🚨
Blocked Until Feasibility Is Confirmed
Srilatha's architectural review of whether ION's current data model can support per-partner filtering is the single biggest blocker for Feature 959113. Until her assessment is complete, engineering cannot proceed.

There's a chicken-and-egg problem: The new pricing model relies on entitlement data from Google to price accurately. But during the purchase (buy) flow, no entitlement exists yet — it's only created after the order goes through.

User places order
→

→

The current recommendation is to show MSRP with a disclaimer during purchase, then reconcile to the correct entitlement-based pricing after the entitlement is created. The downstream reconciliation path for this is still undefined.

End-to-End Pricing Scenarios

Scenario 1 — No pricePhases (Flat Pricing)

A single discountPercentage and discountType apply for the full life of the offer ID under that entitlement. Simple case — Price Book displays these flat values.

Scenario 2 — With pricePhases (Time-Bound Pricing)

Each phase has its own effectivePrice, discount, discountPercentage, and discountType. ION determines which phase is active using firstPeriod values and createTime.

Phase Resolution LogicActive Phase = first phase where: (today − createTime in days) < phase.firstPeriod Example (36-month term): Phase 1: days 1–365 → discountPercentage = 30% Phase 2: days 366–730 → discountPercentage = 20% Phase 3: days 731+ → discountPercentage = 10% If today is day 1097 → Phase 3 applies (10% discount)
Scenario 3 — Seat-Tier Resolution

When seat count changes cross a tier boundary (e.g., Tier 1 = 1–300 seats, Tier 2 = >300 seats), ION must re-evaluate the applicable pricing tier. A move from 300 to 301 seats triggers a tier change and must apply Tier 2 pricing.

Scenario 4 — Buy Flow Gap

No entitlement exists at purchase time. Current recommendation: show MSRP + disclaimer. Resolve exact pricing post-entitlement creation. Downstream reconciliation path not yet documented.

Risk Register · April 2026

Risks & Open Questions

Known risks, key dependencies, and the 14 unresolved questions that must be answered before any workstream can ship.

3 High Risks 4 Medium Risks 1 Low Risk 14 Open Questions

Risk Register

High

Price Book overwrite on every sync

If ION writes the master price book on each GCP sync cycle, reseller overrides are silently erased. No guard exists today.

High

alternateEmail lost on PATCH

Any PATCH to Customer that omits alternateEmail in readMask will clear the field — breaking entitlement lookup silently downstream.

High

Unique ID gap at transfer time

When a subscription transfers from Direct to Reseller, the entitlement ID changes. ION has no reconciliation step — billing continuity breaks for that seat window.

Medium

listEntitlementChanges pagination not implemented

The API returns a nextPageToken for large result sets. ION currently ignores it — change events beyond page 1 are silently dropped.

Medium

discountType ambiguity (PERCENTAGE vs AMOUNT)

ION passes discountPercentage regardless of discountType. If Google returns AMOUNT-type discounts, the stored value is misinterpreted.

Medium

Buy flow bypasses reseller price book

Customers who self-provision via Google's native buy flow see Google list price, not reseller pricing. No mechanism exists to intercept or correct this post-purchase.

Medium

Billing end date formula edge cases

The COMMITMENT_CANCEL_ITEM billing end date formula (start + committed months − 1 day) has not been validated against partial-month commitments or mid-cycle seat changes.

Low

Marketplace link button UX regression

Removing the Marketplace link button (WS6) may confuse admins who bookmarked that navigation path. Low risk because the button links out of ION entirely — removal is straightforward.

14 Open Questions

  1. Should master and reseller price books live in the same DB table with a discriminator column, or in separate tables?
    Context: Separate tables make the sync overwrite problem impossible by design; a single table with a flag is simpler to query but requires careful write guards.
  2. Who is the authoritative owner of the reseller price book — ION or a human operator?
    Context: If ION owns it, we need a UI for reseller margin entry. If a human operator owns it, we need a protected write path that the sync job cannot touch.
  3. What is the expected cadence of GCP price sync, and can ION batch-defer reseller recalculation until after the sync window?
    Context: If sync happens nightly, we can run a post-sync reseller reconciliation job. If sync is real-time, we need row-level locking or an event-driven approach.
  4. Do pricePhases arrays ever contain more than one phase in practice for GWS SKUs?
    Context: The API supports multi-phase pricing, but if GWS SKUs always return a single phase, we can simplify the data model significantly.
  1. Should ION store a mapping table of old entitlement ID → new entitlement ID to handle transfer reconciliation, or rely on the customer's alternateEmail as the durable key?
    Context: The entitlement ID changes on transfer but the customer domain (via alternateEmail) does not. A mapping table adds complexity; using alternateEmail as the key requires it to be reliably present.
  2. Is there a Google API event or webhook that fires when an entitlement transfers, or must ION poll and detect the ID change?
    Context: If a transfer event exists in listEntitlementChanges, WS5 can handle this passively. If not, WS2 needs active polling logic with a diff algorithm.
  1. In the Transfer flow, does the Google API preserve the existing alternateEmail from the previous reseller, or is it cleared and must be re-set by ION?
    Context: If Google clears it, ION must set it as the first call after confirming a transfer — any delay means a window where entitlement lookup fails.
  2. For the "Outside Marketplace" flow (customer exists in Google but not in ION), how does the operator discover the customer's domain to populate alternateEmail?
    Context: This is a pure UX/ops question. The answer drives the design of the manual entry screen in the ION portal.
  3. Is alternateEmail validated by Google (format check, MX record, domain ownership) or accepted as any string?
    Context: If Google validates it, ION must surface API errors meaningfully. If not, ION needs its own validation to prevent garbage data from being stored.
  1. Has the billing end date formula (commitment start + committed months − 1 day) been validated against Google's own cancellation invoice for a real GWS subscription?
    Context: If there's a discrepancy, ION will overbill or underbill for the commitment window. This needs a real test transaction, not just a formula review.
  2. What is the correct credit calculation when a customer cancels mid-commitment and has already been invoiced for the full commitment period?
    Context: The spec mentions a credit, but the formula for proration (daily rate × remaining days vs. monthly chunks) has not been specified.
  1. What is the recommended polling interval for listEntitlementChanges — and does Google impose a rate limit that would constrain a near-real-time implementation?
    Context: Rick Kapani is the DRI for this answer. The polling interval directly determines the maximum delay between a seat change and ION's billing update.
  2. Should ION store the raw listEntitlementChanges payload for audit purposes, or only the derived delta (seats before → seats after)?
    Context: Storing the raw payload enables full replay but adds storage cost. Storing only the delta is leaner but loses context if the interpretation logic changes later.
  3. Is there a defined SLA between a seat change event and when ION must reflect it in billing — and does that SLA differ for increases vs. decreases?
    Context: Seat increases that are billed late undercharge the customer; seat decreases that are billed late overcharge them. The risk profile differs, and the SLA may need to differ too.

Summary

Google now uses the same Offer ID for multiple end customers, but each customer can have different pricePhases and different prices. The current ION Price Book can't distinguish between these customers — when SSIM looks up pricing it either returns 0 or overwrites data incorrectly.

This BRD proposes introducing a Unique Identifier at the entitlement level to allow accurate per-customer, per-phase pricing with day-level granularity. A Master Price Book (ISV) and a filtered Reseller Price Book would be provided. Key constraint: SKU must remain clearly identifiable by the user alongside any new Unique ID fields.

Business Justification

Offer IDs for RABF and standard offers are used across multiple partners and end-customers with varying costs — ION cannot correctly identify the current price for existing subscriptions. The most manual task for TD Synnex Operations is the Price Book price adjust / assignment, which this change directly reduces.

Open Design Questions (from Apr 26–28 email)
  • Filtering: Should offer+entitlement entries for other partners be hidden/filtered on load? Needs SME feasibility discussion.
  • Grouping key: Srilatha proposes discountType; Brian requires visible discountPercentage (e.g., .15) for bulk adjustments. Hybrid approach not yet decided.
  • Auto-inheritance: New Unique IDs added later with matching discountPercentage must automatically inherit the existing price adjustment.

Summary

The alternateEmail field was removed from the buy flow when Feature 755664 (Buy Flow Redesign) shipped on 23 Jan 2026. MSPs need this field to direct Welcome Emails to their own domain instead of the end customer's domain.

Currently ION incorrectly maps the ION Customer Contact Email to Customer.alternateEmail, causing two issues: (1) end customers receive the Welcome email instead of resellers, and (2) the unique email constraint blocks MSPs from reusing a single reseller email across multiple customer registrations.

Key Functional Requirements
  • Register flow: Mandatory alternateEmail field with pre-population if known; user must use Cancel or Confirm to exit
  • Transfer flow: Same field; title renamed from "Link & Transfer Confirmation" to "Transfer Confirmation"
  • Cloud Providers section: View / Edit / Discard / Update for alternateEmail at any time outside buy flow
  • Link button: Remove from Marketplace Plans UI entirely
  • readMask: Must be included in GET Customer calls to surface alternateEmail
Acceptance Criteria
Alternate Email is a mandatory field — user cannot proceed without entering a value
User can only exit edit state via Confirm/Update or Cancel/Discard — clicking away is blocked
System pre-populates Alternate Email when an existing value is known for the customer
Google Welcome email is received at the entered Alternate Email address upon successful provisioning
User can view, edit, and persist Alternate Email from the Cloud Providers section at any time

Summary

When a GWS subscription on an annual commitment plan is cancelled mid-term, the SaaS Report creates a COMMITMENT_CANCEL_ITEM line. The existing rating engine incorrectly calculates Customer Cost — resellers are not receiving the correct credit amount.

Two compounding bugs: (1) BigQuery provides an incorrect billing end date for cancels, and (2) the rating logic used for TERM_START / SEATS usage types does not apply correctly to cancel lines.

The Fix — Formula
  • Discount Factor = 1 − Price Adjust Value
  • Customer Cost = Seller Cost ÷ Discount Factor
  • Example: −493.4 ÷ (1 + (−29.167 ÷ 100)) = −697.78 (correct credit)

Known test cases: Spain (Sep, Dec 2025), UK (Oct, Dec 2025), France (Sep, Nov 2025).

Acceptance Criteria
SaaS line items with charge type COMMITMENT_CANCEL_ITEM use the defined formula to calculate Customer Cost
Calculated Customer Cost matches expected result from formula with no rounding or proration errors
Customer Cost reflects correct credit at both ISV and Reseller level for cancelled subscriptions

Google Workspace Vendor Onboarding into Apptium / ION

This document captures the full context, engineering workstreams, technical logic, and open decisions involved in stabilizing GWS as a vendor in TD SYNNEX's Apptium/ION billing and marketplace platform.

Billing Price Book Entitlement API 7 Workstreams 12 Open Questions

① Core Purpose

🎯
The Single Goal
Every change to a GWS subscription (upgrade, downgrade, plan change, cancellation, seat shift, renewal) must map to a correctly priced line item using the right entitlement context — while resellers see only their own data, and customer alternate email is captured end-to-end.
⚠️
The Central Problem
The existing Price Book architecture was built for flat product catalogs. GWS requires entitlement-level, customer-specific, time-sensitive pricing — a fundamentally different model the current platform doesn't support.

② Main Ideas, Simply Explained

📚
What is a Price Book?
The catalog that tells the system what to charge

Think of a Price Book like a restaurant menu. Today, ION has one shared menu for everyone. GWS requires each table (reseller) to have their own secret menu — with their own customer-specific prices, discounts, and deal terms. The problem: the kitchen (ION) doesn't know how to run that kind of operation yet.

The Master Price Book is what TD SYNNEX/administrators see (full pricing data). The Reseller Price Book is a filtered view — resellers only see their own customers' data, never discount terms or other resellers' info.

🔑
What is an Entitlement?
Google's record that a customer has access to a product

When a customer purchases Google Workspace, Google creates an entitlement — a unique record that says "Customer X has Y seats of Product Z under offer W, at price phase P." This is Google's source of truth for what was sold and when.

ION must read and track these entitlement records to know: what price applies today, what changed over time (upgrades, renewals, seat changes), and what credit is owed on cancellation.

🔢
What is the "Unique ID"?
A new identifier that links pricing to a specific customer's deal

Today ION prices GWS from a generic product catalog entry. The "Unique ID" is a proposed new concept — a composite key that ties a specific entitlement (customer + offer + price phase) to its exact discount percentage. This allows ION to price accurately at the day level across the full life of a subscription.

Think of it as a serial number for a customer's specific pricing deal, not just the product type.

📧
Why does Alternate Email matter?
A lost field that breaks Google customer creation

Customer.alternateEmail is a required field for creating GWS, Cloud Identity, and Chrome Enterprise customers in Google's system. It was removed when Feature 755664 (Buy Flow Redesign) shipped on 23 Jan 2026. Without it, Welcome Emails go to the end customer instead of the MSP/reseller, and the unique email constraint blocks MSPs from reusing a reseller email across multiple customers.

This is classified as a foundational defect — no new GWS customer can be properly onboarded until this field is restored.

Price Phases Discount Logic Entitlement API Unique ID

Key Concepts Explained

A pricePhase is a time-bounded segment of a subscription's pricing. Imagine a 3-year GWS deal: Year 1 might be at 30% discount, Year 2 at 20%, Year 3 at 10%. Each of those is a separate price phase. ION must know which phase is active on any given calendar day to generate correct billing. It does this by taking the subscription's createTime as the start, then counting forward using the firstPeriod value (in days) of each phase.

💡
Mental Model
Think of it like a tiered lease. If you signed a 3-year lease with decreasing discounts, the billing system must know which year you're in — day 364 vs. day 366 can mean entirely different rates.

discountPercentage is the numeric rate (e.g., .15 = 15%). discountType is a category label (e.g., RESELLER_MARGIN, SALES_DISCOUNT). Srilatha's position (Apr 26): Grouping by discountType is the better architectural fit. Any further sub-selection should be done at the row level. Brian's reply (Apr 28): Users must know the exact discount percent to apply the correct PB Price Adjust. Concrete use case: user selects all entries with .15 discountPercentage and bulk-adjusts them. Any new Unique IDs added later with that same .15 discountPercentage must automatically inherit the existing price adjustment.

⚠️
Still Unresolved — Decision Required
The two positions are not necessarily mutually exclusive: a hybrid model (group by discountType architecturally, but surface exact discountPercentage for user selection) may satisfy both sides. No formal decision has been documented yet.

The Unique ID is a proposed new concept in ION. It acts as a composite reference that links one specific customer's entitlement → offer → price-phase context → discount percentage into a single identifiable object.

  • Must resolve to an exact discountPercentage at any point in time
  • Stays stable across seat changes
  • Updates when a SKU_CHANGED or COMMITMENT_CHANGED event fires
  • New IDs with the same discount % must auto-inherit existing bulk price adjustments

Open question: Is this a database-generated surrogate key, or is it derived from Google fields (entitlementID + offerID + startDate)? This isn't decided yet.

When a customer cancels an annual GWS commitment plan, Google generates a COMMITMENT_CANCEL_ITEM charge type in the billing data. ION must issue a credit to the reseller/end-customer for the unused portion. The current bug: ION is using BigQuery's end-date value instead of calculating it independently. BigQuery's value is incorrect, causing wrong credit amounts.

Correct FormulaBilling End Date = createTime (subscription start) + 1 year Credit Calculation: Discount Factor = 1 − Price Adjust Value SaaS Customer Cost = Seller Cost ÷ Discount Factor Example: Seller Cost = −493.4 Price Adjust Value = −29.167% Discount Factor = 1 − (−0.29167) = 1.29167 Customer Cost = −493.4 ÷ 1.29167 = −697.78

This Google API returns a chronological log of every change to a customer's entitlement. ION needs this history to answer: "What was the pricing state on day X of this subscription?"

  • PAID_SUBSCRIPTION_STARTED — the original subscription start
  • SKU_CHANGED — customer switched products
  • COMMITMENT_CHANGED — term length changed
  • RENEWED — annual renewal occurred
  • RENEWAL_SETTING_CHANGED — auto-renew setting changed
⚠️
Critical Open Question
Which createTime is the authoritative start date for billing end-date calculations: the original PAID_SUBSCRIPTION_STARTED, or the most recent RENEWED or SKU_CHANGED? This must be resolved before cancel billing can be correctly implemented.

Currently, a single Price Book in ION can be shared across multiple reseller partners. This means one reseller could potentially see another reseller's discount percentages, customer domain names, and deal terms. The architectural problem: ION currently treats Price Books as completely independent of Partner-Customer associations. Implementing per-partner filtering would require ION to process all Partner-associated customer domains and their entitlements on every Price Book load — a potentially expensive operation.

Brian's Apr 28 clarification: The proposal is to filter/hide offer + entitlement entries belonging to other partners on load. The SKU must remain easily identifiable alongside the Unique ID at all times.

🚨
Blocked Until Feasibility Is Confirmed
Srilatha's architectural review of whether ION's current data model can support per-partner filtering is the single biggest blocker for Feature 959113. Until her assessment is complete, engineering cannot proceed.

There's a chicken-and-egg problem: The new pricing model relies on entitlement data from Google to price accurately. But during the purchase (buy) flow, no entitlement exists yet — it's only created after the order goes through.

The clarified model is one master price book with child price books acting as layered pricing definitions. A child price book is more like "2% on top of parent" than a fully duplicated dataset.

That matters because reseller-specific filtering of entitlement rows does not fit naturally if the child layer is configuration over a parent, not a separate copy of every underlying price record.

RABF refers to special discounted offers created through Google sales flows. These offers may not appear in standard marketplace eligibility, especially when the marketplace is only showing dynamically eligible options for a domain.

They can still appear later in SIM or billing when the customer is linked. Today these are handled through special price book assignment at the cloud-account level, which is why marketplace visibility is not the same as platform support.

The revised model separates Marketplace, SIM, Price Book, and Billing/Reporting into connected systems with different responsibilities.

  • Marketplace handles discovery, eligibility, cart, and purchase initiation.
  • SIM manages subscription and entitlement state synced from the provider.
  • Price Book stores layered pricing configuration and account/entity assignment logic.
  • Billing/Reporting uses provider data plus price-book assignment logic to produce report output.

The proposed pricing key moves from offerId + periodPhase to offerId + entitlement + periodPhase. That may solve the uniqueness problem because the entitlement carries customer-specific subscription context.

The conflict is visibility. If entitlement-level or customer-level data becomes part of price-book rows without a redesigned access model, reseller views may expose information they should not see. The key may be technically useful while still being unsafe inside classic price-book visibility.

Original Spec
Scenario 1 — Flat price / normal catalog assumption

A normal catalog assumes one stable offer-level price can be shown and rated consistently. This is the simple case the existing fixed lookup model handles best.

Transcript
Scenario 2 — Time-bound price phase

Each phase has its own effectivePrice, discount, discountPercentage, and discountType. ION determines which phase is active using firstPeriod values and createTime.

Phase Resolution LogicActive Phase = first phase where: (today − createTime in days) < phase.firstPeriod Example (36-month term): Phase 1: days 1–365 → discountPercentage = 30% Phase 2: days 366–730 → discountPercentage = 20% Phase 3: days 731+ → discountPercentage = 10% If today is day 1097 → Phase 3 applies (10% discount)

When seat count changes cross a tier boundary (e.g., Tier 1 = 1–300 seats, Tier 2 = >300 seats), ION must re-evaluate the applicable pricing tier. A move from 300 to 301 seats triggers a tier change and must apply Tier 2 pricing.

Google vs ION implementation boundary

Full Transcript Transcript FRD 959443 FRD 959440

Google's platform controls eligibility, entitlement identity, and provider-side pricing. ION's price-book layer controls how that pricing is stored, rated, and reported. These are separate concerns — a change in one does not automatically propagate to the other, and most of the GWS-specific problems arise precisely at this handoff boundary.

Summary

The core problem is pricing uniqueness: the same Google offer can have different pricing by customer, commercial arrangement, and period structure. A lookup that only understands offer and phase can return the wrong value.

The proposed key direction is offer + entitlement + period phase. Entitlements are unique for purchases, but the transcript clarification says they were not the original direct association target for price books.

Business Justification

Offer IDs for RABF and standard offers are used across multiple partners and end-customers with varying costs. ION cannot correctly identify the current price for existing subscriptions unless the pricing model understands customer-specific subscription context.

Current Architecture Conflict
  • Proposed direction: expand price identity to offer + entitlement + period phase.
  • Clarification: price books are assigned to members/accounts in the original model, not directly to entitlements.
  • Architectural objection: entitlement rows inside visible price-book data can create reseller visibility and privacy leakage.
  • Open issue: filtering per reseller may not fit the one-master-price-book plus layered child configuration model.
Proposed solutions
Email chainFull TranscriptFRD 959113
  • Entitlement-aware composite key — expand pricing identity to offer + entitlement + period phase to resolve uniqueness for customer-specific pricing arrangements.
  • Filtering on reseller load — filter unrelated partner entries when a reseller loads price-book data, so each reseller sees only rows relevant to their accounts.
  • Grouping by discount type and percentage — consolidate price-book rows that share the same discount structure to reduce duplication and improve display ergonomics.
  • Row-wise bulk adjustment — allow bulk price adjustments across matching price-book rows as an operational shortcut, without changing the underlying identity model.
  • Reseller-limited derived view — introduce a scoped view layer that limits what a reseller sees without altering the master price book structure.
  • MSRP fallback — when an entitlement is absent at buy-flow time, show MSRP with a disclaimer and resolve to actual pricing post-entitlement creation.

Summary

When a GWS subscription on an annual commitment plan is cancelled mid-term, the SaaS Report creates a COMMITMENT_CANCEL_ITEM line. The existing rating engine incorrectly calculates Customer Cost — resellers are not receiving the correct credit amount.

Wrong output example: Seller Cost -493.4 with Price Adjust Value -29.167% should produce Customer Cost -697.78, not the current incorrect output.

Appendix note: BigQuery end-date derivation and data-correction logic for cancel lines is also incomplete, so the fix depends on both rating logic and source-data behavior.

The Fix — Formula
  • Discount Factor = 1 - Price Adjust Value
  • Customer Cost = Seller Cost / Discount Factor
  • Example: -493.4 / (1 + (-29.167 / 100)) = -697.78

Affected countries and billing periods include Spain, UK, and France cases from the FRD validation set.

Acceptance Criteria
SaaS line items with charge type COMMITMENT_CANCEL_ITEM use the defined formula to calculate Customer Cost
Calculated Customer Cost matches expected result from formula with no rounding or proration errors
Customer Cost reflects correct credit at both ISV and Reseller level for cancelled subscriptions
